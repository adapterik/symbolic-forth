Symbolic FORTH

A Different FORTH

Symbolic Forth is a FORTH that reflects both the traditional FORTH approach to programming, yet also brings some concepts and conveniences from my programming career. It is oriented to the browser, so its internals are not integers, pointers and circular stacks, but core Javascript structures like numbers, maps and arrays.

Some types are read in literally and placed directly on stack. These include number, symbol and boolean. Others are stored separately and a reference placed on the stack. These include string, map, and array.

The purpose of this project is education and fun! First, to teach the fundamentals of FORTH, at least at the high level (low level FORTH, the original and most prevalent, this is not) - stacks, reverse polish notation, words, the parser, compiler and interpreter. Second, it is also fun to put this together. That is, it has been fun for me, but I hope it is fun for others to tinker with as well.

Types

FORTH does not traditionally have any sense of types in the modern sense. Or rather, should we say, it is based on the fundametal type being the natural word size of the computer it is running on. There is essentially no type checking. More complex types like doubles, strings, arrays, and so forth do exist, but are typically created and used with bespoke "words" (like functions).

This FORTH is based on browser Javascript so it has some basic types available underneath, and I've chosen to expose these to the language. This allows Symbolic FORTH to be more easily used to program directly against the browser DOM. Yes, it might be fun one day to fork this FORTH and limit it to a more basic type and build from there, but not yet, perhaps never.

Recognizers - Parsing

Recent FORTHs have utilized a technique called "recognizers" to assist in structured, or at least organized, parsing. This project does as well, because it is a handy conceptual model for both partitioning responsibility within the parser, but also for being able to expand it as needed.

A recognizer does what its name suggests - it recognizes something in a FORTH program. It does so at "parsing time" - that is, when the text of the program is being consumed by the FORTH system, and turned into something that can be executed. A recognizer is a small program that, given a text token (essentially a "word" - but not in the FORTH sense of "word") says "aha, I know what that is, I'll handle it!"

This recognition is one way of thinking of parsing. A parser needs to make sense of the text given to it.

Clasically, FORTH can only recognize a token as a number (usually just an integer) and if that fails assumes it is a word and if that fails it considers the program to be in error.

Recognizers expand this mechanism to allow for the recognition of things other than integers, like floating point numbers, symbols, dates, measurements, and so forth. Basically any value that can be represented as a single text token can have a recognizer. In our system, at least, words even have a recognizer. The word recognizer is run last, and if it finds that the token matches a FORTH dictionary word, handles it. If no recognizer is invoked for a token, it is a program error.

FORTH is a word-oriented language, which at the program text level is essentially what we think of as a "word" - continuous letters and numbers without spaces. This "word" is better and less ambiquosly referred to as a "token". FORTH is different than byte-oriented programming languages, in which any character in the text can control the program.

The word/token-oriented nature of FORTH does lead to awkwardness in the case of data or program structures that span across spaces. (I'm still thinking about this, because it seems somewhat arbitrary.) The impact of this is that we can have recognizers for whole word/tokens, like a number, date, symbol, boolean true/false, null, and of course program "words" (aka functions). But we can't have them for strings, arrays, structures. For these, we need to have special words which themselves parse the program text.

So we have token recognizers which interpret a token, but we also have parsing words, which take over the job of parsing the program text after they are themselves recognized as a program word and ending when they sense something to end their parsing in the program text. A string starts with the word ", which then starts eating up the program text until it encounters a literal `"` character in the text, at which point it will stop consuming the program text, create a string with what it consumed, and returns to the main parser.

Built-In Types

This FORTH provides several data types beyond just the integer. In fact, the basic type is the number, not integer, as this is what Javascript offers.

All Symbolic Forth values are stored with their typing intact. This allows us to use polymorphic words when appropriate. For instance, the output word . will print the canonical form of any value provided to it from the Data Stack. However, other types of words like comparisons (equality, less than, etc.) are not polymorphic because the way one compares values of different types are too different. In such cases a generic approach can lead to confusion.

number
A Javascript number - either integer or decimal form accepted, positive or negative. Examples: 123, 1.23, 0, -456
string
A Javascript string, provided in the classic FORTH style, using the double-quote as a word to start reading a string from the program stream. A string ends with the character ". Examples: " a string"
boolean
A Javascript boolean, provided as a raw token of either true or false. Examples: true, false
symbol
This type does not have a Javascript equivalent; it is simply stored as a string. A symbol is basically a single-word string value, with the string stored in a separate symbol table. Within FORTH code it is provided by using a ' prefix. [If FORTH encounters a token which is not a word, number, or boolean, it is interpreted as a symbol. Examples: foo VAR bar, foo .]
array
An array is simply a sequence of any number of FORTH values. We do not currently support words in an array. Arrays may be created, any member accessed, and any member set. We may also get the length of an array.

Vocabularies

Many FORTHs support a "vocabulary" feature. A vocabulary is essentially a namespace (a term used in many programming languages) - a separate storage structure in which words and data may be stored.

Each vocabulary has a name, such as string, used to prefix words available in the vocabulary. For instance string::compare is a word in the string vocabulary which allows one to obtain a string ordering comparison (as implemented in javascript's localeCompare method). Since this word is not relevant to any other data type, it is available only in the string vocabulary.

In fact, this is one of the primary uses of vocbularies - the make type-specific words available in a separate namespace.

Standard(ish) FORTH words are available in the empty vocbulary, which may be accssed without any prefix (or with :: if you like, but that is a bit silly.)

Data Stack Words

Words for manipulating or finding out about the status of the data stack. The data stack is abbreviated to DS.

Note that the Data Stack is also referred to in FORTH systems as the Parameter Stack, as it normal method for a word to receive parameters. At the same time, a word "returns" data by placing it onto the same stack. So, a more general term for this stack seems to be Data Stack.

Return Stack Words

Words for manipulating or finding out about the status of the return stack. The data stack is abbreviated to RS.

In a traditional low level FORTH, the return stack is literally used to place the return address of a word when it, for instance, calls another word. For that reason, although this stack may be used as a general-purpose stack, when it is used within a word (which is everything other than the very top level of FORTH code), upon exit from the word it always must be left in the same state it was in when the word was called.

In this FORTH, the return stack is not used for "return", but functions as second data stack. However, we still do insist that the Return Stack be treated carefully and that it be left in its original state upon a word's exit.

Numeric Comparison

FORTH generally frowns upon words supporting more than one type - in other words, we don't often have polymorphic words. This may make FORTH a bit too literal for some, but it ensures that words have just one meaning, which makes reading and debugging code that much easier.

As a consequence, each of our basic supported types (number, boolean, string, symbol, array, object) has its own set of comparison operators. This section covers just the numeric comparison words.

Each comparison word takes the top two items from the Data Stack, and applies the appropriate comparison. The number to be the subject of the comparison is the second one taken from the stack. That is, when writing a line of code like 1 2 <, we can read this as 1 < 2.

This may seem counterintuitive if you have fully incorporated "stack"-oriented thinking. If you think in stack terms from the point of view of the word itself, the first value to be taken from the stack is the last one put on the stack. So in this sense the first parameter is actually the second one in the code text. However, the textual order, which matches the order in which items are placed on the stack, not taken off of it, is how we read the code.

If either item is not a numeric value, an error message is generated and the forth system resets to its original state. If you wish to debug your program using the state of the system when the error occurs, set the system mode to debug, like 'debug ! MODE.

Variables and Constants

While FORTH is oriented around stacks, and is at its core can be programmed only with stacks, it is rather convenient to be able to store values with an associated name for future and repeated reuse.

Constants and Variables work very similarly, but contstants may not be set more than once.

Both variables and constants work at both compile and real time. This is because variable and constants reference their symbolic name, which must be converted to the equivalent internal identifier. This is equivalent to low level FORTH systems which compile a reference to an address. For the sake of efficiency, this lookup is performed at compile time. In order to accomplish this, the variable name is placed after the word, and the word consumes this token in the program stream during compile time.

One implication of this is that variables must be defined in the system before they are used. This may seem obvious, but there are circumstances in which we want to compile a forth-coded word definition which uses variable references, making this approach a bit tricky. In such cases, variables simply need to be defined before a word using them is defined.

Locals

Local variables are useful in words of more state. Of course, we should attempt to keep words concise and simple.

For clarity, it can useful to populate local variables with parameters, particulary for educational software and for algorithms that would otherwise need to use variables and push and pop values off of the data stack.

Looping

As is the case for conditional words, we have only one looping word at present as well. Of course, we would like to add more, and we may - see the Todo section below.

Word Meaning Compile-time? Run-time? DS used DS left RS used RS left Notes
DO .. LOOP Loops over a section of code N times, where N is determined by two values from the stack as shown above. Yes Yes 2 0 0 0

As with other FORTH words like math operations, for ease of human understanding we discuss the arguments From left to right in textual order, rather in their stack order.

we have a number to count up to - the loop's last run is when the internal counter is the same as count_to. And the second argument is the starting number for the internal counter.

EXIT
BREAK
CONTINUE
RECURSE

Example

0 5 0 DO " iteration: " PRINT DUP PRINTLN 1 + LOOP

Can you guess what this does?

First it puts 0 on the stack - we will use this to count iterations of the loop.

Then it puts 5, the loop limit, and 0, the starting loop count, on the stack.

The DO word consumes the 0 and 5 and uses them internally to control the loop. It proceeds to put a string on the stack, then prints it, removing it from the stack.

The loop code next duplicates the top of the stack, which happens to be the 0 we first put there, and then prints that number and removes it from the stack.

Finally the loop code puts a one on the stack and adds the top two stack items. Underneath the 1 on the stack is our original 0. So now we have incremented the 0 and have 1 on the stack.

You should be able to guess what happens. The code within the loop runs 5 times, printing the numbers 1 through 5. That's it.

BEGIN..REPEAT

The BEGIN word runs a section of code repeatedly. The section of code is terminated by the key-word (token) REPEAT. The mechanism is guided by the value on the stack prior to running BEGIN, and subsequently evaluated in ever repetition. If the value is "truthy" - 1, true, any string other than "", any list with 1 or more elements, any non-empty symbol - the code section is executed; when it becomes "falsey" - any other value - the code section is skipped and the program starts running immediately after the REPEAT key-word.

This is the most general-purpose loop in the language. By decrementing the predecessor value, or any other application logic taht sets the predecessor value, the loop can be controlled arbitrarily. On the other hand, the DO..LOOP always iterates for the given number of times.

Word Meaning Compile-time? Run-time? DS used DS left RS used RS left Notes
BEGIN BEGINs a section of code, terminated by REPEAT, which is repeated as long as the top stack value is truthy. Yes Yes 1 0 0 0

Example

3 BEGIN " Count: " PRINT DUP PRINTLN 1 - REPEAT

Note that it is common to structure code textually as above - with the code section indented and framed by the BEGIN and REPEAT. But this is FORTH, and all that matters is whitespace.
