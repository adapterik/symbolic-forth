Symbolic FORTH

A Different FORTH

Symbolic FORTH is one that reflects the traditional FORTH approach to programming, yet operates at a "higher level", lacking much of the low-level machinery of classic, traditional FORTH, but bringing a lot of other functionality.

It is implemented in Javascript, so is well-suited to programming in a web browser through the browser's built-in javascript engine as well as on the command line through standalone javascript implementations like QuickJS. As a result its internals are not integers, pointers and circular stacks, as in low-level FORTHS, but core Javascript structures like numbers, maps and arrays. In addition, Javascript itself has an extensive API for manipulating its types, constructions programs, etc. so our foundation is already much more functional than Assembly language or plain C - the basis for most FORTH implementations.

How is Symbolic FORTH Similar to other FORTHS?

I link to think of FORTH as an abstract virtual machine - there are certain properties of a FORTH system that I consider fundamental. Symbolic FORTH exists in that abstract virtual machine. In sum, here is what I mean:

  • It is stack based: It relies upon a parameter stack for arguments to and output from words (like functions)
  • It supports a return stack, but does not use it for anything itself, it is just available as a useful programming tool
  • It is concatenative in that the program flow is strictly linear.
  • The language itself parses and compiles - it enables words to both parse, compile, and execute at run time
  • The fundamental tokens recognized by the parser are only integers and words; parsing words provide all of any specialized forms like numeric forms, arrays, maps, dates.
  • Supports variables, constants, locals, loops, colon-words

How is Symbolic FORTH different from other FORTHS?

As much as I like to think of Symbolic FORTH as a real FORTH, it does differ by omission and addition:

Omissions

  • It does not utilize or allow direct memory access
  • It uses native Javascript data types to implement all of its own data types
  • The core wordset is larger than many minimal FORTHs, due to the fact that we cannot (or have not) implemented more sophisticated features requiring raw memory access - e.g. structures, arrays, floats.

Additions

  • Vocabularies are used extensively
  • Locals are supported and used, when appropriate, liberally.
  • The basic numeric type is a JS "number" - which can be treated as a float or integer
  • Rather than using addresses to point to data in memory (as opposed to the stack), non-stack data is stored in global structures and accessed by identifier.
  • Types are supported and stored with all FORTH values
  • Built in types include number, string, symbol, map, array, date, boolean, null

Symbolic FORTH Features

These are some of the things I consider special about Symbolic FORTH

Runs in the browser or command line

Has testing framework - minimal but works in browser and command line

Useful built-in types

Easy to extend with Javascript

Locals and type-specific words - e.g. L:>[ spreads array elements into local variables

In Progress or To Do

Error handling is undecided - exceptions need to be handled because JS may throw them, but whether we want to rely upon them or use value-based errors is something to be decided and considered

Colon-words do not currently support parsing, though Javascript-defined ones do.

Functionality is added as needed - so stuff that may be in a fully implemented FORTH are surely missing.

Loading of vocabularies is not supported in FORTH code itself

Etc.

Some types are read in literally and placed directly on stack. These include number, symbol and boolean. Others are stored separately and a reference placed on the stack. These include string, map, and array.

The purpose of this project is education and fun! First, to teach the fundamentals of FORTH, at least at the high level (low level FORTH, the original and most prevalent, this is not) - stacks, reverse polish notation, words, the parser, compiler and interpreter. Second, it is also fun to put this together. That is, it has been fun for me, but I hope it is fun for others to tinker with as well.
